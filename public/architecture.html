Auth Architecture — How It Works

▶

Authentication Flow (6 Steps)

1

Supabase Auth Issues JWT

User signs up at murphy-research-ai.vercel.app (email/password). Supabase returns a signed JWT (HS256) containing user_id, email, and an expiry. User sees their token + QR code on the dashboard page.

2

Token Paired to Field Unit

User opens Murphy UI on iPad at FOH (http://192.168.1.50:8000). Auth gate shows pairing screen — scan QR code or paste token manually. Token stored in localStorage and sent as Authorization: Bearer on all API calls.

3

Local JWT Validation (Air-Gap Safe)

Field unit validates JWTs locally via HMAC-SHA256 signature check. No network call. JWT secret from Supabase stored in settings table. Verifies: (1) signature valid, (2) not expired, (3) sub claim present. Works fully air-gapped during shows.

4

FastAPI Dependencies Check JWT + Role

All 74 endpoints protected via get_current_user, require_engineer, require_admin FastAPI dependencies. First-seen users auto-cached in users_cache with default viewer role. Dev mode bypass available (auth_mode=dev).

5

Role-Based Access Control

Three roles: admin (full access + settings + WiFi), engineer (mixing + OSC + channels), viewer (read-only + chat). Admin can promote users via /api/auth/users/{id}/role.

6

Channel Identity Linking

The channel_identities table (deployed on Supabase) maps a single user_id to multiple platform accounts. Table exists but linking logic not yet built. When complete, Tommy on Discord and Tommy on WhatsApp resolve to the same engineer profile.

Sync Protocol — Tailscale (Between Shows Only)

↓ Pull First (Mothership → Field)

  • JWT refresh + public key rotation
  • Updated venue profile for upcoming show
  • Latest model weights (if new version)
  • Gear subset for current band
  • LanceDB vector subset (active context)
↓ TAILSCALE ↑

↑ Then Push (Field → Mothership)

  • Override queue (raw corrections)
  • Training pairs generated since last sync
  • Completed show metadata
  • Correction signals (gig_corrections)
  • Sync log entry (entity_type, count, timestamps)

Ownership Rules

  • Field unit owns show-time data
  • Mothership owns identity + catalog
  • Catalog changes only flow between shows
  • Conflict: most-recent timestamp wins
  • Never blocks during a live show
